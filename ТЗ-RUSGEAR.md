# Техническое задание — интернет-магазин «RUSGEAR»

> Статус: черновик v0.1 (по итогам интервью). Открытые вопросы — в разделе 12.
> Дата: 2026-07-13.

---

## 1. Общее описание проекта

**RUSGEAR** — интернет-магазин бронежилетов и средств индивидуальной бронезащиты.

- **Ниша:** бронежилеты, бронеэлементы, тактическое снаряжение.
- **Ценовой сегмент:** средний и премиум.
- **Модель продаж:** B2B-опт (основной фокус), каталог и цены открыты всем посетителям.
- **Объём каталога:** 50–500 товаров.
- **Источник контента:** магазин продавца на OZON —
  https://www.ozon.ru/seller/rusgear/ (фото и описания берём оттуда).

### 1.1 Архитектура решения (важно)

Проект строится **не с нуля**. Есть готовая платформа-бэкенд **Admik** (репозиторий
`Uni_BD`, выкачан в `rusgear/Uni_BD`):

- headless-бэкенд + админка + БД PostgreSQL + публичный **Storefront API**
  (`/api/storefront/v1/*`);
- встроенные модули: каталог, заказы/корзина, промо/акции, СДЭК (доставка),
  оплата Т-Банк, CMS, настройки/брендинг, формы/подписка;
- стек: **Next.js 16 (App Router) + React 19 + Tailwind CSS + TypeScript**,
  PostgreSQL, Redis, S3/MinIO.

**Задача проекта** — спроектировать и собрать **макет витрины (storefront) на Next.js**,
который затем «прикручивается» к платформе Admik: витрина становится тонким
потребителем Storefront API и **не содержит собственной бизнес-логики и БД**.
Все цены, остатки, расчёты доставки и оплаты считаются на сервере (anti-tamper).

Модель развёртывания платформы: **1 магазин = 1 БД + 1 VPS**; тема магазина задаётся
конфигурацией (`.env` + `shop_settings`) и данными, без правок кода платформы.

## 2. Целевая аудитория и сценарии

- **Кто покупает:** оптовые закупщики (юрлица), силовые/охранные структуры, дилеры;
  плюс розничные покупатели среднего/премиум сегмента.
- **Поведение:** осознанный выбор и сравнение (класс защиты, вес, комплектация),
  а не импульсная покупка.
- **Приоритет устройства:** **Desktop-first** (закупщики работают с ПК), с
  обязательной адаптивностью под планшет/мобильные.
- **География:** Россия (доставка СДЭК; город отправления — уточнить).

## 3. Дизайн-система

### 3.1 Стилистика
- **Строгий минимализм.**
- Референсы: **tactec.ru** (нравится сетка/отступы от краёв, стиль карточек товара,
  шрифты) и **allmulticam.ru**.

### 3.2 Цвета
- **Основной:** серый (нейтральная база интерфейса).
- **Фирменные дополнительные:** `#3A301F` (тёмный), `#B5A07F` (светлый).
- **Акцентные:** `#21519E` (синий), `#CE3730` (красный — CTA/скидки/важное).
- Цвета доезжают до витрины через `GET /settings → branding.theme`
  (`primaryColor`, `accentColor`, `mode`).

### 3.3 Типографика
- **Заголовки:** HALVAR BREITSCHRIFT.
- **Основной текст:** MONTSERRAT.
- Дополнительные шрифты не вводить.

### 3.4 Сетка и компоненты
Единая сетка с выраженными отступами от краёв (по мотивам tactec.ru).
Переиспользуемые компоненты:
- карточка товара (с бейджами скидка/новинка);
- фильтры каталога;
- навигация (шапка/футер);
- кнопки CTA;
- модальные окна;
- формы;
- бейджи (скидка, новинка, класс защиты).

## 4. Карта страниц (витрина)

| Страница | Назначение | Источник данных (Storefront API) |
|---|---|---|
| Главная | Витрина: hero, преимущества, доставка, промо | `GET /settings` (`home`, `navigation`, `branding`), `GET /promotions`, `GET /products?featured=1` |
| Каталог / Категория | Список товаров + фильтры + поиск | `GET /products` (q, category, brandId, new, sale, featured, limit/offset), `GET /categories`, `GET /brands` |
| Карточка товара (PDP) | Описание, характеристики, варианты, медиа, SEO | `GET /products/:slug` |
| Корзина | Предпросчёт цен/скидок/доставки | `POST /cart/quote` |
| Оформление заказа (Checkout) | Данные покупателя, доставка, оплата | `POST /orders`, СДЭК (`cities`/`pvz`/`calculate`), оплата |
| Личный кабинет | Регистрация/вход, история заказов, избранное, адреса/реквизиты | **доработка платформы** (см. §7) + `GET /orders/:number?token=` |
| Трекинг заказа | Статус заказа | `GET /orders/:number?token=` |
| Избранное | Список отложенных товаров | **доработка** (привязано к аккаунту, §7) |
| Поиск | Результаты по запросу | `GET /products?q=` |
| Акции / Скидки | Активные акции | `GET /promotions` |
| Контентные страницы | О компании, доставка, оплата, оферта, контакты | `GET /pages`, `GET /pages/:slug` |
| Формы | Обратная связь, подписка | `POST /leads`, `POST /newsletter` |

## 5. Каталог: структура и данные

- **Категории:** дерево (`GET /categories`) — сформировать по ассортименту с OZON.
- **Варианты товара:** **размеры/ростовки** (у части товаров), с отдельными
  остатками (`product_variants` + `inventory`). Есть и **простые товары без вариантов**.
- **Характеристики (фильтры):** EAV-атрибуты — как минимум класс защиты, материал,
  вес, бренд. Итоговый список — при разборе OZON.
- **Поля товара:** название, цена, старая цена (`compareAtPrice`), скидка,
  бейджи new/sale/featured, бренд, медиа, описание, SEO-мета.
- **Вес/габариты** обязательны для товаров/вариантов — используются для расчёта СДЭК.
- **Деньги** приходят в копейках; в рубли конвертирует витрина по `settings.currency`.

## 7. Доработки платформы (расширения Admik)

> Реализуются аддитивно, по TDD, в `app/api/storefront/v1/*` + `lib/*`, без ломки
> мультитенантности. Каждая — потенциально переиспользуемый модуль для будущих магазинов.

1. **Оплата «СДЭК PAY».** Платформа из коробки даёт **Т-Банк**; выбрано добавить
   **СДЭК PAY** как модуль оплаты (оплата и доставка в одном контуре СДЭК).
   *Решение:* новый payment-модуль по аналогии с `payments/tbank/*`
   (init → redirect на paymentUrl, webhook статуса). Отразить в чекауте и трекинге.
2. **Личный кабинет покупателя (полноценный).** Регистрация/вход, профиль, история
   заказов, сохранённые адреса и **реквизиты юрлица** (для B2B). Требует модуля
   авторизации покупателей (в текущем API заказы — гостевые, трекинг по номеру/токену).
3. **Избранное (wishlist), привязанное к аккаунту.** Хранение на сервере и
   синхронизация между устройствами (зависит от п.2).
4. **Уведомления в Telegram-бот** о заказах/событиях (в платформе есть только
   webhook-алерты мониторинга — не пользовательские уведомления о заказах).
5. **E-mail уведомления о заказах** (в платформе из штатного — только чек 54-ФЗ
   на стороне провайдера оплаты).
6. **Интеграция с CRM** (система — не выбрана, см. §12): выгрузка заказов/клиентов.

## 9. Оплата и B2B-условия

- **Цены видны всем** посетителям (не скрываются под авторизацией).
- **Способ оплаты:** СДЭК PAY (доработка). Для B2B желательно предусмотреть
  оплату по счёту для юрлиц — **уточнить** (см. §12).
- Anti-tamper: суммы заказа и доставки считает сервер; витрина передаёт только
  выбор покупателя (id/qty/промокод/адрес/способ).

## 10. Технические ограничения и стандарты

- **Стек витрины фиксирован платформой:** Next.js 16 (App Router) + React 19 +
  Tailwind + TypeScript (strict). Витрина отдаёт `output: 'standalone'`, порт 3000,
  свой Dockerfile; кладётся в `./storefront` рядом с платформой.
- Витрина — **без собственной БД и бизнес-логики**; данные только из Storefront API.
- Два адреса API: SSR → `http://app:3000` (внутр. сеть, ключ `X-Storefront-Key`);
  браузер → `https://${ADMIN_DOMAIN}` (по Origin).
- Медиа — готовые публичные URL (через Caddy `/media/*`).
- Обязательные проверки платформы: `pnpm typecheck` (0) · `pnpm lint` (0) ·
  `pnpm test` (vitest) + живой прогон в браузере (Playwright).

## 11. Этапы работ (по методологии платформы, docs/20)

1. **Профиль магазина из макета** (read-only анализ) → карта данных.
2. **Интервью владельца** (частично проведено — этот документ).
3. **Поднятие БД + миграции + seed** (`make up → init → smoke`).
4. **Ретематизация под нишу:** `.env`, каталог/атрибуты, брендинг (`shop_settings`),
   CMS-страницы.
5. **Сращивание витрины с Storefront API.**
6. **Две проверки (код-гейт + браузер) + документирование.**

Дизайн-часть (макет витрины): Interview → Analysis → UX Structure → Page Map →
UI System → Design Execution → Documentation → Handoff.

## 12. Открытые вопросы (нужно от владельца)

1. **CRM-система** — какая (Битрикс24 / amoCRM / 1С / другое)? Пока не выбрана.
2. **Оплата по счёту для юрлиц** (B2B) — нужна ли в дополнение к СДЭК PAY?
3. **Город/склад отправления** для расчёта СДЭК.
4. **Реквизиты юрлица** (наименование, ИНН, КПП, ОГРН, юр. адрес) — для чеков/футера.
5. **Домены:** витрина и админка (для деплоя и CORS).
6. **Юридическая часть** (сертификаты, классы защиты, ограничения продажи/вывоза) —
   отложено, вернуться перед боевым запуском.
7. **Структура категорий и список атрибутов-фильтров** — финализировать при разборе
   каталога с OZON.
8. **Тексты страниц** (О компании, доставка, оплата, оферта, политика) — источник/копирайт.
9. **Режим запуска:** демонстрационный (mock-интеграции) или боевой (реальные ключи).
