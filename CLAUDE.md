## 1. Что это за проект

**RUSGEAR** — интернет-магазин бронежилетов и средств бронезащиты.
Сегмент: средний/премиум. Модель: **B2B-опт** (цены открыты всем). Каталог: 50–500 товаров.
Контент (фото/описания) — с OZON: https://www.ozon.ru/seller/rusgear/

**Мы делаем макет витрины (storefront) на Next.js.** Бэкенд уже готов — это платформа
**Admik** в подпапке [`Uni_BD/`](Uni_BD/) (админка + БД PostgreSQL + Storefront API +
СДЭК + оплата + CMS). Витрина — **тонкий потребитель** `/api/storefront/v1/*`, **без
собственной БД и бизнес-логики**. Все цены/остатки/доставку/оплату считает сервер.

- Контракт API: [`Uni_BD/docs/21-контракт-storefront-api.md`](Uni_BD/docs/21-контракт-storefront-api.md)
- Плейбук сращивания: [`Uni_BD/docs/20-онбординг-витрины-playbook.md`](Uni_BD/docs/20-онбординг-витрины-playbook.md)
- Правила платформы: [`Uni_BD/CLAUDE.md`](Uni_BD/CLAUDE.md)

## 2. Приоритет работ (ВАЖНО)

**Фаза 1 — сначала простой макет.** Доп. функции подключаем позже.

Витрина в первой фазе (desktop-first, строгий минимализм):
- Главная (см. §4)
- Каталог / страница категории (список, фильтры, поиск)
- Карточка товара (PDP)
- Корзина
- Оформление заказа — **гостевое, базовое** (без регистрации)
- Контентные страницы (О компании, доставка, оплата, контакты)

**Фаза 2 — отложено (доработки платформы, не сейчас):**
- Полноценный личный кабинет покупателя (регистрация/вход, история, реквизиты)
- Избранное, привязанное к аккаунту
- Оплата **СДЭК PAY** (в коробке платформы — Т-Банк; СДЭК PAY = отдельный модуль)
- Telegram-уведомления о заказах
- E-mail уведомления о заказах
- Интеграция с CRM

> В Фазе 1 не проектируем экраны ЛК/избранного/регистрации. Оплату в макете
> показываем на штатном Т-Банк / mock; СДЭК PAY заводим в Фазе 2.

## 3. Дизайн-система

- **Стиль:** строгий минимализм. Референсы: **tactec.ru** (сетка/отступы от краёв,
  карточки товара, шрифты — эталон), **allmulticam.ru**.
- **Устройство:** **desktop-first**, обязательная адаптивность.
- **Цвета:**
  - основной — серый (нейтральная база интерфейса);
  - фирменные — `#3A301F` (тёмный), `#B5A07F` (светлый);
  - акцентные — `#21519E` (синий), `#CE3730` (красный: CTA, скидки, важное).
- **Шрифты:** заголовки — **HALVAR BREITSCHRIFT**; текст — **MONTSERRAT**.
  Других шрифтов не добавлять.
- **Компоненты (переиспользуемые):** карточка товара (+бейджи скидка/новинка),
  фильтры, навигация, кнопки CTA, модалки, формы, бейджи.
- Брендинг/цвета/навигация приезжают с сервера: `GET /settings → branding/navigation/home`.

## 4. Главная страница (блоков немного)

Порядок сверху вниз:
1. **Шапка** — логотип, меню категорий, поиск, корзина. *(избранное/ЛК — Фаза 2.)*
2. **Главный блок (hero)** — имиджевый баннер с CTA.
3. **Хиты товаров** — `GET /products?featured=1`.
4. **Новинки** — `GET /products?new=1`.
5. **Блок доверия/преимуществ** (доставка, гарантия, качество) — коротко, по `settings.home`.
6. **Подвал** — контакты, меню, подписка (`POST /newsletter`). *(реквизиты — позже.)*

> Держать главную лаконичной: не плодить блоки. Промо-баннер акций
> (`GET /promotions`) — опционально, если появятся активные акции.

## 5. Маппинг «страница → Storefront API»

| Что на витрине | Эндпоинт |
|---|---|
| Список товаров / каталог + фильтры | `GET /products` (q, category, brandId, new, sale, featured, limit/offset) |
| Карточка товара (варианты/медиа/SEO) | `GET /products/:slug` |
| Категории (меню/навигация) | `GET /categories` |
| Бренды | `GET /brands` |
| Предпросчёт корзины | `POST /cart/quote` |
| Оформление заказа | `POST /orders` (сохранить `accessToken`) |
| Трекинг заказа | `GET /orders/:number?token=` |
| Доставка СДЭК: город/ПВЗ/расчёт | `GET /delivery/cdek/cities`, `/pvz`, `POST /delivery/cdek/calculate` |
| Оплата | `POST /payments/tbank/init` → редирект (СДЭК PAY — Фаза 2) |
| Контентные страницы | `GET /pages`, `GET /pages/:slug` |
| Брендинг/меню/главная/контакты | `GET /settings` |
| Акции-баннер | `GET /promotions` |
| Обратная связь / подписка | `POST /leads`, `POST /newsletter` |

**Anti-tamper:** НЕ отправлять с витрины цены/итоги/вес — сервер их игнорирует и
считает сам. Витрина передаёт только выбор (id/qty/промокод/адрес/способ).
**Деньги — в копейках**, конвертирует витрина по `settings.currency`.

## 6. Каталог: данные

- Категории — дерево (`GET /categories`), собрать по ассортименту с OZON.
- **Варианты:** размеры/ростовки (у части товаров) + **простые товары без вариантов**.
- **Фильтры-атрибуты:** класс защиты, материал, вес, бренд (финализировать по OZON).
- Вес/габариты товара обязательны — нужны для расчёта СДЭК.

## 7. Технический стек и правила (заданы платформой)

- **Next.js 16 (App Router) + React 19 + Tailwind CSS + TypeScript (strict).**
- Витрина: `output: 'standalone'`, порт 3000, свой Dockerfile; кладётся в `./storefront`
  рядом с платформой (см. `docker-compose.yml` платформы).
- Без собственной БД/бизнес-логики; данные только из Storefront API. Локально можно
  работать в mock-режиме API (пустые ключи/origins → доступ открыт).
- Два адреса API: SSR → `http://app:3000` (ключ `X-Storefront-Key`); браузер →
  `https://${ADMIN_DOMAIN}` (по Origin).
- Медиа — готовые публичные URL, дополнительное проксирование не нужно.
- Проверки перед коммитом: `pnpm typecheck` (0) · `pnpm lint` (0) · `pnpm test` +
  живой прогон в браузере (Playwright). Сверять API библиотек через MCP context7, не по памяти.

## 8. Открытые вопросы (заполнить позже, старт не блокируют)

- **CRM** — система не выбрана.
- **Оплата по счёту для юрлиц** — не нужна.
- **Город/склад отправления** СДЭК — нет данных.
- **Реквизиты юрлица** (ИНН, КПП, ОГРН, юр. адрес) — нет данных.
- **Домены** витрины/админки — настроим позже.
- **Тексты** страниц (О компании, доставка, оферта…) — черновики придумываем сами,
  финал — от владельца.
- **Юр. часть** (сертификаты, классы защиты, ограничения продажи) — вернуться перед
  боевым запуском.
